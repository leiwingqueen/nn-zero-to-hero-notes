BigramLanguageModel 图解

整个模型只有一个 nn.Embedding(vocab_size, vocab_size)。 没有注意力、没有隐藏层——当前 token 的 id 直接索引出下一个 token 的 logits。 这是后续所有 Transformer 改进要击败的基线模型。

B batch，几段文本一起训练
T time / block，每段文本的长度
C channel，这里等于 vocab_size

02

forward(idx, targets) · 从 id 到 loss

前向传播只做两件事：查表得到 logits； 如果给了 targets，就把 (B,T) 拍平成 (B*T,)，用 交叉熵算出一个标量 loss。

idx (B, T) int
一批字符 id，来自训练数据的随机切片
token_embedding_table(idx) 查表
每个 id 独立查出它自己的那一行
logits (B, T, C)
C = vocab_size，每个位置一份"下一个字符"的打分
targets is None
loss = None

推理 / 生成阶段——只要 logits

targets 已给出
.view(B*T, C) / .view(B*T)
logits 与 targets 各自拍平成二维 / 一维，交给 cross_entropy
F.cross_entropy → loss

训练阶段——衡量预测分布与真实下一个字符的差距

F.cross_entropy 内部在算什么
Step 1 · softmax 归一化
pi,j = elogits[i,j] / Σk elogits[i,k]
Step 2 · 取真实标签对应概率，取负对数
Li = −log pi, y(i)
Step 3 · 对 B·T 个位置取平均
loss = (1 / B·T) · Σi Li

对照第 01 节的例子：输入 'h' 时，logits 里 'e' 那一维最大。 如果真实的下一个字符恰好是 'e'，softmax 后它的概率会接近 1，−log(≈1) ≈ 0，loss 很小； 反之若模型预测错了，正确标签对应的概率很低，−log(≈0) 会变得很大——这就是"惩罚"预测错误的方式。

实现细节：PyTorch 的 F.cross_entropy 不会真的先算 softmax 再取 log， 而是用 log_softmax + nll_loss 一步完成（内部走 log-sum-exp 技巧）， 避免 e^logits 数值上溢或下溢。输入必须是未归一化的原始 logits， 不能提前 softmax 过——这是最容易踩的坑。

03

generate(idx, max_new_tokens) · 自回归采样循环

生成时模型每次只关心最后一个时间步的 logits， 转成概率后按概率随机采样一个新字符，拼回序列末尾，再整体喂回模型——如此循环。

1
logits, _ = self(idx) (B, T, C)
把目前为止的整段序列重新跑一遍前向传播
2
logits[:, -1, :] (B, C)
只保留最后一个位置——它预测的是"接下来"那个字符
3
F.softmax(logits, dim=-1) (B, C)
logits 转成一个合法的概率分布
4
torch.multinomial(probs, 1) (B, 1)
按概率随机采样，而不是直接取最大值——所以每次生成结果都不同
5
torch.cat((idx, idx_next), 1) (B, T+1)
采样出的新字符拼到序列末尾，序列变长 1

重复以上 5 步，直到生成满 max_new_tokens 个字符为止

这就是 m.generate(idx=torch.zeros((1,1)), max_new_tokens=100) 背后发生的全部事情—— 没有上下文窗口、没有注意力，每一步的预测只依赖上一个字符。 这正是接下来引入 self-attention 要解决的局限。
